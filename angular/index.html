AngularJS: Overview

What is Angular

Angular is a JavaScript framework that is designed to ease the creation of highly dynamic websites.

The Old World


<input type="text" id="input" value="" name="name">
<p>Hello <span id="name"></span></p>
<script type='text/javascript'>
window.onload = function() {
  var input = document.getElementById('input'),
      nameBox = document.getElementById('name');
  input.addEventListener('change', function() {
    nameBox.innerHTML = input.value;
  });
};
</script>

Controllers

Controllers

A controller allows you to manipulate the state and data associated with the view without having any of the overhead of DOM manipulation.

How To Define A Controller


// you need at least a reference to your application
// or the module in your applicaiton that you are working on
angular.module('ea')
  // to add a controller you then use the .controller method
  // the .controller method takes the controller name as the first
  // parameter, and then the callback, or the function that defines
  // the controller as the second
  .controller('ControllerNameCtrl', function($scope) {
    // we are not going to go into detail about dependency injection
    // here, or scope, we will get into that later
  });

$scope

$scope is the shared context between controllers and views. You can access any variables or methods defined in the $scope in the view.

Data Binding

Angular offers one way and two way data binding. In two way data binding, the value in the view can update the value in the controller, and vice-versa. In one way data binding, the value in the controller can update the value in the view.

Directives

Directives

Directives are perhaps the most important feature of angular. Directives allow for code reuse in a very convienient way.

plnk me

How To Define A Directive


angular.module('ea')
  // to add a directive you then use the .directive method
  // the .directive method takes the directive name as the first
  // parameter, and then a function that returns the directive definition
  // Directive names are normalized
  .directive('ToDoList', function() {
    return {
      restrict: 'AEC',
      scope: {},
      controller: 'ToDoListCtrl',
      link: function(scope, elem, attrs) {},
      template: 'hello'
    };
  });

restrict

restrict lets angular know how this directive can be used, as a tag name (E), class name (C) or an attribute (A). In Origin X, we use tag name directives heavily, but attribute directives are extremely important. If your directive has a template, you should probably use a tag name. If your directive adds behaviour, use an attribute.

scope

You can choose to have the inherit values from its parent, or have an isolated scope.

Isolated Scope

An isolated scope inherits from $rootScope (the top most scope), and does not have any inherited values. This means that in order to pass information down to the scope of the directive, we have to explicitly do so using the scope option in the directive declaration.

scope


scope: {
  // this binds the value of the attribute
  // text binding to the scope variable textBinding
  textBinding: '@textbinding',
  // this binds the variable referenced by
  // the attribute oneway to the scope variable
  // oneWay, but, as the name says, it only goes
  // one way
  oneWay: '&oneway',
  // like oneWay, this binds the variable referenced
  // by the attribute twoway to twoWay, but there is
  // two way data binding
  twoWay: '=twoway'
}

template

A directive can have a template, that is, some markup associated with it. If the template is small, you can inline it into the directive declaration using template. If the template is larger, put it into its own html file and reference it using templateUrl. If a directive has a template, I suggest using a element name based directive.

controller

A directive can optionally have a controller associated with it. The value for the controller attribute in the directive declaration is the name of the controller. In Origin X, almost all directives have controllers. In the controller, we place all of our data for the view, as well as any view based logic.

link

A link function gets executed after the element has been compiled. Notice that you get a reference to a scope, the element, and the attributes. Though this is a bit confusing, the best practice is to put any DOM manipulations into the link function of a directive, and put any state based logic into the controller.

Transclusion

Transclusion

In Origin X, we nest our components, so that there is a heirarchy of components. This enables us to plug and play components easily, and create cool new layouts with data driven from the CMS. In fact, it enables us to create an XML representation of our application with data embedded, that in the future can be used as a middle layer to build the application in any language!

plnk!

Transcluded Scope

When you transclude directives, a transcluded scope is created. This scope inherits from its parent, and applies to the transcluded elements of the directive. However, if the directive has an isolated scope, then the directive itself will have an isolated scope, but not its transcluded children (they have the transcluded scope)
